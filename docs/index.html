Pulse

Edge-aware apps, without the transport ritual.

Pulse is a small runtime for realtime flows, bounded compute endpoints, and deploy-time sanity. It keeps the runtime clean, smooths repeated workflow friction, and stays explicit where it matters.

Why Pulse

Most teams keep re-solving the same ugly seams:
  • local vs production drift
  • assets across local, hosted, and bucket-backed deploys
  • bind/listen differences between Node and fetch-style runtimes
  • realtime route behavior leaking socket and proxy mechanics into app code
Pulse makes those seams smaller.
Execution model

Strict enough to compile

Pulse uses a narrow execution contract that makes lowering possible. A working app can be translated from AST, lowered, and emitted as a compact executable target for environments like Node and Fastly.

When a path falls back to TS/JS, the debt stays visible. You are not punished for shipping — you are shown where the contract broke.

Execution model

Structural truth under pressure

Pulse keeps the fast path narrow on purpose. That strictness is what makes AST lowering, specialization, and compact compiled targets possible. When you need to ship outside the contract, the system still lets you move — but it keeps the fallback visible.

  • Whole-engine lowering A working app can be translated from AST, lowered, and emitted as a compact executable target instead of staying interpretive.
  • Fallback debt stays explicit Pulse is unforgiving in the right way. It shows you where execution had to be propped up by TS/JS instead of pretending the fast path still held.
  • Extension points without drift Pulse-aware plugins can provide their own lowering path, and specialized parser / serializer sidecars can stay outside the main binary when that keeps the target cleaner.

What the contract buys you

Predictable lowering, compact targets, and a brutally legible boundary between what stayed eligible for compilation and what did not.
